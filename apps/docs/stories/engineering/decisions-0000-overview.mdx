<Title>Architecture Decision Records</Title>

This section documents the **technical decisions** behind Shark UI — not just
_what_ the system does, but _why_ it was built this way, which alternatives were
weighed, and what each choice costs.

Each page is an **ADR** (Architecture Decision Record): a short, dated,
immutable note capturing one decision at the moment it was made. It is the
lightweight standard for recording architecture rationale, and it keeps the
reasoning next to the code instead of in someone's head.

## Format

Every record follows the same shape:

- **Status** — Accepted / Superseded / Deprecated.
- **Context** — the forces at play: the problem, the constraints, what was true
  when the call was made.
- **Decision** — the choice, stated plainly.
- **Alternatives considered** — the options that were rejected, and why.
- **Consequences** — what this buys, and the trade-offs it accepts.

An ADR is a snapshot. It is not edited to reflect later reality — if a decision
changes, a new record supersedes the old one, so the history stays honest.

   `apps/docs/stories/engineering/`.
2. Title it `Engineering/Decisions/NNNN · <short title>` — the numeric prefix
   keeps the sidebar ordered.
